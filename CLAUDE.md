# CLAUDE.md — Permafrost Operating Manual

This file is the operating manual for anyone (human or Claude session) working on Permafrost.
Read it before changing anything.

## Mission

Bring the Windows `Win + V` clipboard experience to macOS as a fast, native, local-first,
privacy-respecting menu bar app — with a better retention model: **pinned entries never
expire; unpinned entries expire after a configurable period.**

## Goals

- Instant: panel opens in < 100 ms, search filters as you type
- Native: SwiftUI/AppKit, correct light/dark behavior, feels like part of macOS
- Keyboard-first: every action reachable without the mouse
- Trustworthy: local-only storage, no telemetry, honors concealed pasteboard types
- Ownable: an engineer who has never seen this repo can build, test, and extend it in an afternoon

## Non-goals

- A productivity suite. No launcher, no snippets-with-templating, no window management.
- Cloud sync of any kind (see docs/FUTURE_IDEAS.md — even there it is treated with suspicion)
- Supporting macOS < 14
- Mac App Store distribution (sandbox forbids the paste simulation that defines the product — ADR-007)

## Coding philosophy

- Maintainability over cleverness. Boring code that obviously works beats smart code.
- Native APIs over dependencies. The dependency budget is **one**: GRDB (ADR-003).
  Adding any dependency requires an ADR.
- Small modules with one job. `PermafrostCore` (headless, tested) must never import AppKit.
- No premature optimization. Measure with Instruments before optimizing.
- No TODO comment without a matching item in docs/BACKLOG.md.

## Architecture philosophy

- `PermafrostCore` (library): models, storage, retention, search, import/export. Pure, testable.
- `Permafrost` (executable): AppKit shell, pasteboard watcher, hotkey, panel UI, settings.
- UI observes core; core never knows UI exists.
- All persistence goes through `ClipboardStore`. No SQL outside `PermafrostCore`.

## Documentation standards

- Docs live in `docs/` and are updated **in the same commit** as the change they describe.
- `docs/DECISIONS.md` holds ADRs. Numbered, never deleted; superseded ADRs are marked as such.
- An ADR is required when: adding/removing a dependency, changing the storage schema,
  changing distribution/signing strategy, changing the permission model, or reversing a prior ADR.
- `docs/BACKLOG.md` is the single source of truth for deferred work.

## Testing expectations

- Everything in `PermafrostCore` has unit tests (Swift Testing, `swift test`).
- Retention logic must have explicit tests: pinned survives purge; unpinned expires; caps enforced.
- UI/permission flows are covered by the manual checklist in docs/TESTING.md — run it before tagging.
- CI (`.github/workflows/ci.yml`) must be green before merge to `main`.

## Commit standards

- Conventional Commits: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`, `chore:`, `ci:`.
- Subject ≤ 72 chars, imperative mood. Body explains *why* when non-obvious.
- Each commit is a working state (builds + tests pass).

## Branch strategy

- Trunk-based: `main` is always releasable.
- Short-lived feature branches (`feat/panel-ui`), merged via PR or fast-forward for solo work.
- Tags: `vMAJOR.MINOR.PATCH` (SemVer).

## Definition of Done

A change is done when:

1. Code builds (`swift build`) and tests pass (`swift test`)
2. New logic in core has tests
3. Docs affected by the change are updated in the same commit
4. Deferred work is captured in docs/BACKLOG.md
5. If it changed an architectural decision, an ADR exists
6. The manual smoke test relevant to the change passes (docs/TESTING.md)
7. If it completes a ROADMAP.md milestone, `PermafrostVersion.string` (App.swift) and
   `VERSION` (scripts/make-app.sh) are bumped together, and README.md's Status line
   matches — this has drifted silently twice already (see ADR history in DECISIONS.md);
   check `grep -rn "PermafrostVersion.string\|VERSION=" Sources scripts` before tagging

## Pull request expectations

- One logical change per PR. Description says what and why, links issues.
- Screenshots/recordings for UI changes.
- No red CI. No unresolved review threads.

## Security principles

- No network code. Adding any networking is a project-redefining decision requiring an ADR
  and a very good reason.
- Never record pasteboard content marked `org.nspasteboard.TransientType` or
  `org.nspasteboard.AutoGeneratedType`. Content marked `org.nspasteboard.ConcealedType`
  (passwords) is skipped **by default**; recording it is a user opt-in gated behind an
  explicit risk-acknowledgment prompt (ADR-011, docs/SECURITY.md). Opted-in items get the
  full feature set — visible, searchable, pinnable — marked with a 🔑 indicator.
- Database and support directory are owner-only (`0700`/`0600`).
- No analytics, no crash uploaders. `os.Logger` only; never log clipboard *content*.

## Performance goals

- Panel open: < 100 ms perceived
- Search keystroke → filtered list: < 16 ms at 5,000 items
- Idle CPU: ~0% (pasteboard poll every 0.3 s is a changeCount integer compare)
- Memory: < 50 MB resident with text-only history; image blobs stay on disk until displayed

## Decision-making framework

1. Does it serve the mission (Win+V, fast, private)? If not, it's FUTURE_IDEAS.md or rejected.
2. Is there a native API? Use it. A dependency needs an ADR proving the native path is worse.
3. Will a future maintainer understand it? If it needs a comment to explain *what*, rewrite it.
4. Uncertain? Research first (docs/RESEARCH.md pattern), decide, record the ADR, move on.

## How future Claude sessions should work

1. Read this file, then docs/PROJECT_PLAN.md (current state) and docs/BACKLOG.md (what's next).
2. Work autonomously through backlog items; commit at logical milestones.
3. Keep docs synchronized in the same commit as code.
4. Never leave the repo in a non-building state.
5. When you make a non-trivial choice, add an ADR — future sessions can't read your mind.
6. Update docs/PROJECT_PLAN.md status when a milestone completes.

## Rules for updating documentation

- Code change that alters behavior → update README/UX.md as applicable, same commit.
- New deferred idea → docs/FUTURE_IDEAS.md (product ideas) or docs/BACKLOG.md (engineering tasks).
- ADRs are append-only. To reverse one, write a new ADR that supersedes it and cross-link both.

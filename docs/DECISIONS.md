# Architecture Decision Records

ADRs are append-only. To reverse one, add a new ADR that supersedes it and cross-link both.
Format: Context → Decision → Consequences. Status is Accepted unless noted.

---

## ADR-001: Build a new app instead of adopting or extending Maccy

**Context.** Research ([RESEARCH.md](RESEARCH.md)) showed Maccy (MIT, active, 20k+ stars)
covers ~90% of requirements. It lacks the core retention semantic (pinned = forever,
unpinned = time-based expiry — an open, unimplemented feature request upstream) and uses a
list UI rather than a Win+V panel. macOS Tahoe's Spotlight history is text-only, caps at
7 days, and cannot pin.

**Decision.** Build Permafrost. Maccy is the recommended stopgap daily driver during
development and serves as an MIT-licensed reference implementation. Contributing time-based
retention upstream remains a complementary option, not a replacement for this project.

**Consequences.** We own maintenance forever. Scope discipline is mandatory — the moment
this becomes "another productivity suite" the justification for existing collapses.

## ADR-002: SwiftUI views inside an AppKit shell (not pure SwiftUI, not pure AppKit)

**Context.** The Win+V panel must appear *without stealing focus* from the target app, at a
controlled position, and close on Esc/focus-loss. SwiftUI's `MenuBarExtra` cannot do a
focus-preserving free-floating panel; pure AppKit makes the list UI needlessly laborious.

**Decision.** AppKit owns the app lifecycle, status item, and a borderless
`.nonactivatingPanel` `NSPanel`; SwiftUI (via `NSHostingView`) owns everything drawn inside
it. Keyboard routing for ↑↓⏎Esc⌥P⌘1-9 uses a local `NSEvent` monitor in the panel controller
(reliable regardless of SwiftUI focus quirks).

**Consequences.** Two UI idioms in one codebase, but each does the only thing it's good at.

## ADR-003: SQLite via GRDB + FTS5 for persistence (the one allowed dependency)

**Context.** Options: SwiftData (immature, macOS-14 churn, no FTS), Core Data (no FTS,
heavy ceremony for a two-table schema), raw sqlite3 C API (error-prone plumbing), GRDB
(mature, MIT, typed records, migrations, first-class FTS5 on the system SQLite).

**Decision.** GRDB. This is the project's **single third-party dependency** and the
documented exception to the native-first rule. Search uses an FTS5 external-content table
synchronized by triggers.

**Consequences.** Retention is one indexed `DELETE`; search is one FTS query; migrations are
explicit. Any future dependency proposal must clear the same bar this one did.

## ADR-004: Poll `NSPasteboard.changeCount` (0.3 s) for capture

**Context.** macOS has no pasteboard-change notification API. Every clipboard manager polls.

**Decision.** A main-actor timer compares `changeCount` every 0.3 s; only on change does it
read contents. Captures skip `org.nspasteboard.TransientType` and `AutoGeneratedType`
always, and `ConcealedType` unless the user has opted in (ADR-011).

**Consequences.** Idle cost is an integer compare ~3×/s (unmeasurable). Worst-case capture
latency 0.3 s, imperceptible for this use case.

## ADR-005: Carbon `RegisterEventHotKey` for the global hotkey

**Context.** Native options: CGEvent tap (needs Accessibility just to *listen*, overkill),
`NSEvent.addGlobalMonitor` (cannot consume the event), Carbon hotkey API (deprecated-ish but
fully supported, used by virtually every Mac utility). Third-party recorder libraries exist
but violate the dependency budget for MVP.

**Decision.** Carbon `RegisterEventHotKey` behind a small Swift wrapper. Default `⌥⌘V`
(deliberately not `⇧⌘V`, which collides with "Paste and Match Style"). Settings offers a
preset list; a full custom recorder is backlogged.

**Consequences.** ~80 lines of well-understood code, zero dependencies.

## ADR-006: Paste-on-select via CGEvent ⌘V synthesis + Accessibility permission

**Context.** "Select → it pastes" is the essence of Win+V. The only way to paste into
another app is synthesizing ⌘V, which requires the user to grant Accessibility.

**Decision.** `PasteService` writes the item to the pasteboard, then posts ⌘V key events.
If Accessibility isn't granted: one-time onboarding alert with a deep link to the settings
pane, and graceful fallback to copy-only mode (panel selection loads the pasteboard; user
presses ⌘V themselves).

**Consequences.** One permission ask, clearly explained. The app is fully usable without it.

## ADR-007: No App Sandbox; direct distribution (no Mac App Store)

**Context.** Sandboxed apps cannot post CGEvents to other processes — paste-on-select is
impossible under the sandbox. MAS also forbids the ad-hoc/deferred-signing dev model.

**Decision.** No sandbox. Distribution: build-from-source now; Developer ID signing +
notarization + Homebrew cask at v1.0 (requires Apple Developer Program, $99/yr — deferred
per project owner's decision, personal use first).

**Consequences.** MAS is permanently off the table (recorded as a non-goal). Privacy claims
rest on open source + no-network-code, verifiable in the repo.

## ADR-008: No at-rest encryption in MVP; rely on FileVault + file permissions

**Context.** Full-DB encryption (SQLCipher) adds a dependency and a key-management story;
field-level encryption breaks FTS5 search. macOS FileVault already encrypts at rest, and the
DB directory is `0700`/files `0600`. The highest-value privacy control is *not recording*
sensitive content in the first place (concealed types, ADR-004).

**Decision.** MVP: FileVault + owner-only permissions + concealed-type exclusion. Optional
CryptoKit blob encryption (key in Keychain, search limited to metadata for encrypted items)
is backlogged with a design sketch in FUTURE_IDEAS.md.

**Consequences.** Honest, documented threat model (SECURITY.md) instead of encryption
theater. Revisit if users demand protection against other-admin-user access.

## ADR-009: Plain SPM + bundle script; no Xcode project

**Context.** The dev machine has Command Line Tools only (no Xcode). SPM builds SwiftUI/
AppKit apps fine; what Xcode adds (Interface Builder, asset catalogs, notarytool) is not
needed for MVP. A checked-in `.xcodeproj` is merge-conflict bait and would not even build here.

**Decision.** Pure SPM package (`PermafrostCore` library + `Permafrost` executable).
`scripts/make-app.sh` assembles `Permafrost.app` (Info.plist with `LSUIElement`, binary,
ad-hoc codesign). CI = GitHub Actions macOS runner running `swift build && swift test`.

**Consequences.** Anyone with CLT can build. Xcode users can still `open Package.swift`.
Notarization tooling arrives with the v1.0 distribution work.

## ADR-010: Images/screen snips are MVP; expiry measured from `last_used_at`

**Context.** Project owner: "screen snips are vital" (Win+Shift+S parity). And for expiry,
`created_at` would delete an entry the user pastes daily.

**Decision.** (a) Capture `public.png`/`public.tiff` pasteboard content (normalized to PNG),
store original + downscaled thumbnail, cap configurable (default 10 MB, oversized items
skipped). (b) Retention cutoff compares `last_used_at`; copy-again and paste both bump it.
OCR/search-inside-images stays post-MVP.

**Consequences.** DB can grow with heavy snip use — mitigated by TTL, count cap, and size
cap. "Recently useful" items survive; genuinely stale ones thaw and expire.

## ADR-011: Recording passwords is an informed opt-in, not a hard block

**Context.** Default industry behavior (and ADR-004's default) is to never record
`ConcealedType` content. The project owner explicitly wants the *option* to store copied
passwords — visible in the list and pinnable — accepting the risk knowingly.

**Decision.** Settings → Privacy → "Record concealed content (passwords)", default **off**.
Enabling it presents a risk-acknowledgment dialog (full text in SECURITY.md); Cancel
reverts the toggle. Once acknowledged, concealed items are first-class entries — listed,
searchable, pinnable, exported — marked with a 🔑 indicator in the panel. Items are flagged
`concealed` in the schema so a future "purge all concealed items" action and the indicator
are possible. Transient/auto-generated types remain skipped unconditionally.

**Consequences.** Users who opt in trade safety for convenience with eyes open; the
default remains safe. Export archives may contain secrets — SECURITY.md says so plainly.

## ADR-012: Pinned section moves to the bottom; quick-paste digits address only recent items; bulk actions gain "Unpin All"

**Context.** First real-world use (project owner, 2026-07-06) surfaced a genuine design
flaw: with pinned items sorted to the *top* (the original M2 ordering), pinning something
displaced the most recent copy from the front of the list and, worse, from the `⌘1`
quick-paste slot — pressing `⌘1` after pinning an old item pasted the pinned item, not the
newest copy. The owner's read: pinned items are things reused *over time*, not necessarily
the most recently used thing, so they don't belong competing with "what did I just copy"
for the top slot or the fast-paste numbers. Separately, the only bulk-history action was
"Clear Unpinned" — there was no fast way to demote everything pinned back to normal history
without deleting it outright.

**Decision.**

1. `ClipboardStore.items()` now orders unpinned entries first (by recency), with pinned
   entries in their own section strictly after (most-recently-pinned first). This is a
   product guarantee, not a display detail: `⌘1`–`⌘9` in `PanelModel.commitQuickPaste`
   compute their bound from the live unpinned prefix, so a number key can never paste a
   pinned item under the belief it's the latest copy.
2. Pin section position and quick-paste scoping are **not configurable** — this is the one
   correct behavior, not a preference toggle, matching the project's stance against
   settings-surface creep (CLAUDE.md coding philosophy).
3. Added `ClipboardStore.unpinAll()` — converts every pinned row back to unpinned in place
   (no deletion). Combined with the existing `clearHistory(keepPinned:)`, the status-bar
   menu and Settings → History Management now expose three tiers: Clear Unpinned History…
   (safe to pinned), Unpin All Items… (safe, non-destructive — pinned content survives,
   just loses its expiry exemption), Clear Everything… (destructive, both). All three stay
   out of the panel's keyboard surface deliberately — the panel is optimized for speed, and
   a bulk-destroy shortcut there is an accident waiting to happen.

**Consequences.** Existing ordering tests and the ARCHITECTURE.md schema note were updated
to match (recent-first, pinned-at-bottom). A store-level test
(`unpinnedAlwaysPrecedePinnedRegardlessOfRecency`) protects the invariant the UI-layer
quick-paste logic depends on, since the app target has no automated test harness of its
own (executable is a thin shell over `PermafrostCore` by design).

## ADR-013: Status item image set to template mode; explicit isVisible

**Context.** Project owner reported "my menu isn't opening" after a routine rebuild
(2026-07-06/07). Investigation (see docs/TESTING.md → Scripted verification for the tooling
detour this took) confirmed via `os.Logger` that `setupStatusItem()` completes normally —
`statusItem.button` is non-nil, the SF Symbol image loads, and `isVisible` reads `true` —
yet no snowflake icon was visible in a genuinely-revealed (non-fullscreen) menu bar.
Separately, `⌥⌘V` was confirmed, via a real screenshot showing actual captured clipboard
history in the panel, to work correctly — the core feature was never broken, only the
status-bar icon's visibility was in question. The original code
(`NSImage(systemSymbolName:accessibilityDescription:)` assigned directly to
`statusItem.button?.image`) never set `isTemplate`, meaning the OS was never told to treat
it as a monochrome icon that adapts to the menu bar's own tinting/vibrancy — a known class
of bug where a status item can be fully present in the object model while rendering as
effectively invisible against certain menu bar backgrounds.

**Decision.** Set `image?.isTemplate = true` before assigning it, and set
`statusItem.isVisible = true` explicitly rather than relying on the default. Added
temporary-but-retained `Log.app.info` calls around status item creation (button-nil check,
image-load check, isVisible check) since this class of bug produces no error path to catch
otherwise — the object model reports success at every step regardless of what's actually
on screen.

**Consequences.** This is recorded honestly as a **plausible, applied fix, not a
visually-confirmed one** — a post-fix screenshot still didn't unambiguously show the icon,
though that screenshot session was also fighting an unrelated menu-bar-visibility/Control
Center-aggregation investigation (see docs/TESTING.md) that made positive identification
hard regardless of whether the fix worked. `isTemplate = true` is correct regardless of
whether it was the root cause — it's how every other well-behaved status item image on
macOS is supposed to be configured — so it stays either way. If the icon is still not
visible after this, the next place to look is whether `NSStatusBar.system` is silently
declining to render additional items when the bar is already crowded (no overflow chevron
was observed, but behavior here varies by macOS version) — tracked in docs/BACKLOG.md.
